# 그래프 기반 단원 추천 & 오답 원인 진단 서비스 기획안 (Draft)

## 1) 배경과 목적

현재 서비스 방향은 다음 두 가지를 동시에 해결하는 것이다.

1. **학생의 다음 학습 단원을 추천**한다.
2. 학생이 문제를 틀렸을 때 **정확한 원인(왜 틀렸는지)**을 진단한다.

핵심은 단순 추천 정확도보다, **추천/진단의 근거를 설명 가능하게 만드는 것**이다.

---

## 2) 문제 정의

기존 학습 서비스에서 자주 발생하는 문제:

- 추천이 “왜 이 단원인지” 설명되지 않아 납득이 어렵다.
- 오답 분석이 “틀림/맞음” 수준에 머물러 재학습 연결이 약하다.
- 약점 태그가 초기 진단 1회성으로 끝나면, 실제 학습 변화가 반영되지 않는다.

---

## 3) 서비스 컨셉

## **A. 단원 그래프 (Learning Path Graph)**

- 노드: 단원/스킬
- 엣지:
  - `requires`: 선수 개념(선행 필요)
  - `prepares_for`: 다음 학습 추천 연결

역할:

- 학습 가능한 단원(AVAILABLE) 판별
- 다음 단원 후보 생성

## **B. 오답 원인 그래프 (Error Cause Graph)**

- 오답을 원인 유형으로 분류:
  - `개념 부족`
  - `절차 실수`
  - `계산 실수`
  - `독해/조건 해석 실수`
- 문제 ↔ 필요 스킬 ↔ 대표 오개념을 연결

역할:

- “왜 틀렸는지”를 구조적으로 설명
- 어떤 선수개념 보강이 필요한지 자동 연결

> 결론: **단원 그래프가 ‘어디로 갈지’를 정하고, 오답 원인 그래프가 ‘왜 돌아가야 하는지’를 설명한다.**

---

## 4) 추천 엔진 설계 아이디어

다음 단원 추천 점수를 아래처럼 합성한다.

```text
추천점수 = w1*그래프가능성 + w2*학습효율 + w3*약점보정 + w4*난이도적합도
```

- **그래프가능성**: prerequisites 충족 여부 (`requires` 기반)
- **학습효율**: 최근 정답률/풀이시간/재시도율
- **약점보정**: 최근 weak tag와 단원 태그의 일치도
- **난이도적합도**: 현재 추정 레벨 대비 과난도/과저난도 패널티

추천 결과는 최대 1~3개로 제한하고, 각 추천에 근거를 함께 표시한다.

예시:

- “분수의 덧셈: 최근 관련 태그 오답률이 높고, 선수개념 2개가 충족되어 다음 학습으로 적합합니다.”

---

## 5) 오답 원인 진단 설계 아이디어

문항 제출 시 아래 순서로 진단한다.

1. 문항별 정오/풀이시간/수정횟수 수집
2. 문항이 요구하는 스킬 노드 매핑
3. 오답 패턴 규칙(또는 모델) 적용
4. 원인 카테고리 + 근거 생성
5. 재학습 단원/문항 자동 추천

진단 결과 포맷 예시:

```json
{
  "problemId": "P-1023",
  "isCorrect": false,
  "causeType": "개념 부족",
  "relatedSkills": ["분수 통분", "최소공배수"],
  "evidence": [
    "유사 유형 최근 5문항 중 4문항 오답",
    "선수개념 노드 미완료 1개"
  ],
  "nextAction": "분수 통분 기초 3문항 재학습"
}
```

## 6) 사용자 경험(UX) 원칙

### 학생 화면

- “다음에 할 것 1~2개”만 선명하게 제시
- 각 추천에 **한 줄 근거** 제공
- 오답에는 **원인 + 바로가기 학습** 제공

### 교사/운영 화면

- 학생별 원인 Top 3
- 반/그룹 단위 공통 취약 개념
- 재학습 후 개선률(오답→정답 전환률)

---

## 7) MVP 우선순위

### Phase 1 (빠른 검증)

1. `weakTagsTop3`를 진단 1회성에서 **실시간 갱신형**으로 전환
2. 오답 원인 4분류 규칙 도입(룰 기반)
3. 추천 결과에 근거 문장 자동 생성

### Phase 2 (정교화)

1. 문항-스킬-오개념 매핑 확장
2. 추천점수 가중치 실험(A/B)
3. 교사 대시보드(원인/개선률)

### Phase 3 (고도화)

1. 학생 군집별 개인화 정책
2. 베이지안/IRT 기반 숙련도 추정 도입 검토
3. 설명 가능한 모델 기반 진단 자동화

---

## 8) 핵심 지표(KPI)

- **추천 수용률**: 추천 단원 클릭/시작 비율
- **학습 전환률**: 추천 시작 후 완료 비율
- **재도전 개선률**: 오답 후 재도전 정답 전환율
- **약점 개선 속도**: weak tag 점수 하락 속도
- **설명 신뢰도**: 학생/교사의 “근거 납득” 피드백

---

## 9) 리스크와 대응

- 리스크: 원인 진단이 과도하게 단순화될 수 있음  
  대응: 원인 카테고리는 단순하게 시작하되, 근거 이벤트 로그를 반드시 함께 저장

- 리스크: 추천이 너무 많아 선택 피로 발생  
  대응: 추천 개수 상한(1~3개), 1순위 명확화

- 리스크: 초기 데이터 품질 부족(오개념 매핑 미비)  
  대응: 상위 빈출 단원부터 점진 매핑, 운영 피드백 루프 구축

---

## 10) 한 줄 결론

이 서비스의 승부처는 **“무엇을 추천할지”보다 “왜 이 추천인지, 왜 틀렸는지”를 설득력 있게 보여주는 것**이다.  
즉, **그래프 기반 추천 + 설명 가능한 진단**을 함께 설계해야 한다.
