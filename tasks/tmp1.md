좋아. **“2번(현재 구현된 기능 기준으로 Done/Partial/To‑do로 재정렬한 실행 백로그)”** 형태로 정리해줄게.

다만 한 가지는 투명하게 말할게: **이 채팅 환경에서는 GitHub/네트워크 접근이 막혀 있어서 저장소를 직접 pull/열람해서 ‘코드로’ 확인은 못 했어.**
그래서 아래 상태 분류는 **(1) 네가 말한 구조(`curriculum-viewer`, FE/BE, docs 존재), (2) 다른 리뷰어가 실제로 본 화면 피드백(트리/그래프/리포트, 즉시 채점, 사이드 패널 등)**을 근거로 “현재 구현 상태를 최대한 그럴듯하게” 분류한 거야.
→ 그래서 **첫 번째 티켓을 ‘레포 실사(Audit)’로 잡아서 Done/Partial을 확정**하도록 구성했어.

# 실행 백로그 v1 (Done / Partial / To‑do 재정렬)

## 상태 표기 규칙

* **DONE**: 리뷰어 피드백상 이미 화면에서 확인된 기능(존재 확률 매우 높음)
* **PARTIAL**: 기능은 있으나 MVP 요구사항과 다름(수정 필요)
* **TO‑DO**: 리뷰어가 “없다”고 지적했거나 MVP 요구사항상 필수인데 흔적이 없는 것
* **VERIFY**: “있을 수도/없을 수도”라 Audit로 확정해야 하는 것

---

## EPIC 0) 레포 실사(Audit) + MVP 기준 재정렬 (P0)

> 이걸 먼저 해야 “꼬임”이 사라져. PM/개발자가 1~2시간 안에 끝낼 수 있는 수준으로 쪼갰어.

### MVP-A00. 현재 구현 기능 인벤토리 체크리스트 작성 및 결과 기록

* **우선순위**: P0
* **상태**: TO‑DO
* **목표(Why)**: Done/Partial 판단을 “추측”이 아니라 “팩트”로 고정
* **작업(What)**

  * FE 실행: 라우트 목록, 페이지(트리/그래프/리포트/학습) 스크린샷 저장
  * BE 실행: API 목록(swagger 등) 캡처
  * 진행 저장 위치 확인(LocalStorage/DB/파일)
* **DoD**

  * “기능 목록 + 구현 위치(파일/폴더) + 스샷” 문서 1개 생성
  * 이 문서 기준으로 아래 백로그의 DONE/PARTIAL/TO‑DO를 확정 업데이트

### MVP-A01. 현재 ‘즉시 채점’ → ‘일괄 제출 채점’ 변경 영향도 분석

* **우선순위**: P0
* **상태**: TO‑DO
* **목표**: UX/데이터모델 변경 범위를 확정해서 되돌림 방지
* **DoD**:

  * “어디서 채점하는지(프론트/백), 제출 데이터 구조, 결과 저장 구조”가 정리됨

### MVP-A02. “학생 모드 vs 개발자(Author) 모드” 화면/권한/라우트 정리

* **우선순위**: P0
* **상태**: TO‑DO
* **DoD**:

  * Author 전용 기능(그래프 편집/검증/Publish)와 Student 기능 분리 표가 나온다

## EPIC 1) 학생 계정/진행 데이터(클리어 상태의 근간) (P0)

### MVP-010. 로그인/세션 유지(학생)

* **우선순위**: P0
* **상태**: VERIFY
* **근거**: 리뷰어 피드백에 로그인 언급 없음(있을 수도, 없을 수도)
* **목표**: “로그인 시 클리어 노드 색상” 요구사항 충족
* **DoD**:

  * 로그인 후 새로고침해도 유지
  * 사용자별 진행 데이터 분리됨

### MVP-090. 노드별 진행 데이터 모델(상태/정답률/최근학습)

* **우선순위**: P0
* **상태**: TO‑DO
* **근거**: “트리/그래프에서 학습 진행 상태 표시 부재” 지적
* **DoD**:

  * nodeId별로 `status(CLEARED/AVAILABLE/LOCKED/IN_PROGRESS)`를 반환/계산 가능
  * nodeId별 `accuracy`, `lastAttemptAt` 최소 저장

### MVP-073. 클리어 판정 로직(≥80% 등) + 저장

* **우선순위**: P0
* **상태**: VERIFY (채점은 있으나 “클리어/잠금해제”로 이어지는지는 불명)
* **DoD**:

  * 제출 결과에 따라 해당 노드가 CLEARED로 저장됨
  * 지도에서 즉시 반영됨

## EPIC 2) 학생 지도(트리/그래프) 상태 표시 + 잠금해제 규칙 (P0)

### MVP-040. 지도/트리/그래프 렌더 + 노드 상세 패널

* **우선순위**: P0
* **상태**: DONE
* **근거**: “트리/그래프/리포트 네비”, “상세 패널” 강점으로 언급됨
* **DoD(유지 조건)**:

  * 노드 클릭 시 사이드 패널 정보 표출이 안정적으로 동작

### MVP-0303. 지도에서 완료/진행/미완료/잠김 상태 시각화(색/흐림)

* **우선순위**: P0
* **상태**: TO‑DO
* **근거**: 리뷰어가 “학습 진행 상태 표시 부재”를 명시적으로 지적
* **DoD**:

  * CLEARED(완료 색) / AVAILABLE(도전 가능 강조) / IN_PROGRESS / LOCKED(흐림) 표시
  * 범례(legend) 제공

### MVP-0304. LOCKED 노드 클릭 시 “왜 잠겼는지(선행 노드)” 안내

* **우선순위**: P0
* **상태**: TO‑DO
* **DoD**:

  * 잠김 노드 클릭하면 “필수 선행 노드 목록”이 표시됨

### MVP-0305. “도전하기” 버튼 활성 규칙 적용

* **우선순위**: P0
* **상태**: TO‑DO
* **DoD**:

  * AVAILABLE/IN_PROGRESS만 도전 버튼 활성
  * LOCKED는 버튼 비활성 + 이유 표시

## EPIC 3) 문제 풀이 UX(주관식 + 일괄 제출 채점) (P0)

### MVP-0404. 문제 풀이 화면(주관식 입력 폼, 고정 순서)

* **우선순위**: P0
* **상태**: DONE(화면은 있음) / PARTIAL(정책이 다름)
* **근거**: “트리에서 주제 선택 → 학습 페이지” 강점으로 언급됨
* **필수 수정**: “즉시 채점” → “모두 풀고 제출”
* **DoD**:

  * 문제 순서 고정
  * 입력은 모두 주관식
  * 제출 전엔 정오/정답 노출 없음

### MVP-0406. 제출 API(일괄 채점) + 결과 저장(SUBMITTED)

* **우선순위**: P0
* **상태**: PARTIAL
* **근거**: “실시간 채점” 존재(채점 자체는 있음)
* **해야 할 변경**: 제출 시점에만 채점/결과 생성
* **DoD**:

  * “제출” 버튼 누른 후에만 결과 생성
  * 제출 후 결과가 저장되고 재진입해도 동일 결과(새로고침 안정)

### MVP-0403. 답안 임시저장(DRAFT) + 이어풀기

* **우선순위**: P0
* **상태**: VERIFY
* **DoD**:

  * 입력 중 나갔다가 돌아오면 복원됨(최소 LocalStorage라도)

### MVP-0405. 미입력 문항 안내 + 제출 확인 모달

* **우선순위**: P1
* **상태**: TO‑DO
* **DoD**:

  * “3,7번 비었습니다” 안내
  * 제출 확인 모달(실수 방지)

# 최종 정리: “지금 상태 기준” 실행 순서(P0만)

PM이 바로 스프린트 백로그를 만들 때는 아래 순서가 제일 안전해.

1. **MVP-A00/A01/A02 (Audit로 Done/Partial 확정)**
2. **진행 데이터(노드 상태) 구축**: MVP-090, MVP-073
3. **지도 상태 시각화 + 잠김 이유**: MVP-0303/0304/0305
4. **즉시 채점 → 일괄 제출 채점 전환**: MVP-0404/0406 (+ 임시저장 0403)
5. **평가 페이지 강화 + 다음 스텝 이동**: MVP-0502/0504/0505
6. **대시보드 추가 + 네비 메뉴**: MVP-0601/0602
