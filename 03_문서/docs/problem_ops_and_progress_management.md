# 문제 출제 · Railway 배포 · 조교 운영 · 학생 진도체크 운영안

## 0. 문서 목적
이 문서는 `calculate_math`를 실제 운영하는 관점에서 다음 네 가지 흐름을 하나의 체계로 정리한다.

- 문제를 출제하고 숙제 세트를 만든다.
- 만든 숙제를 Railway 기반 운영 환경에 반영한다.
- 조교가 제출 여부와 검토 상태를 관리한다.
- 학생이 특정 개념(예: 2차함수)을 이해했는지 지속적으로 추적한다.

이 문서는 단순 기능 목록이 아니라, **누가 무엇을 책임지고 어떤 데이터와 화면이 필요하며 어떤 순서로 운영할지**를 정의하기 위한 운영 기준 문서다.

---

## 1. 한 줄 정의

**calculate_math의 운영 핵심은 “문제 출제 → 숙제 배포 → 제출 확인 → 오답/진도 해석 → 다음 액션”이 끊기지 않게 만드는 것이다.**

즉, 이 서비스는 문제를 많이 쌓는 것보다도,

- 숙제가 실제로 배포되고,
- 학생이 제출했는지 확인되며,
- 어떤 개념이 해결되었는지 추적되고,
- 그 결과가 다음 숙제/추천으로 이어지는 운영 구조

를 먼저 가져야 한다.

---

## 2. 운영 관점의 핵심 가치

### 2.1 출제의 가치
- 문제는 단순 콘텐츠가 아니라 **학생 상태를 읽기 위한 측정 도구**다.
- 각 문제는 가능한 한 스킬/단원과 연결되어야 하며, 나중에 “무엇을 몰랐는가”를 해석할 수 있어야 한다.

### 2.2 배포의 가치
- 출제만 하고 운영 환경에 반영되지 않으면 숙제 루프가 끊긴다.
- 따라서 출제 이후 Railway 반영은 부가 작업이 아니라 **운영의 필수 단계**다.

### 2.3 조교 운영의 가치
- 학생이 제출했는지, 아직 안 했는지, 반려되었는지, 완료되었는지 확인하는 과정은 학습 유지율에 직접 연결된다.
- 조교는 단순 체크 인력이 아니라 **학습 루프를 닫아주는 운영자**다.

### 2.4 진도체크의 가치
- “2차함수를 몰랐다”는 사실만으로는 충분하지 않다.
- 중요한 것은 이후 숙제/문제풀이/피드백을 거친 뒤, **그 개념이 해결되었는지**를 다시 판단하는 것이다.
- 이 진도체크가 있어야 추천과 진단도 실제 학습 개선으로 연결된다.

---

## 3. 운영 플로우 전체 구조

### 3.1 기본 플로우
1. 문제 출제
2. 문제/숙제 세트 구성
3. Railway 운영 환경 반영
4. 학생에게 숙제 노출
5. 학생 제출
6. 조교 제출 여부 및 검토 상태 확인
7. 오답/약점/진도 상태 업데이트
8. 다음 숙제 또는 다음 추천 생성

### 3.2 핵심 원칙
- 한 번의 숙제는 **운영 이벤트**여야 한다.
- 숙제는 반드시 추적 가능한 상태를 가져야 한다.
  - 배정됨
  - 제출함
  - 검토중
  - 완료
  - 반려/재제출 필요
- 특정 개념은 반드시 **전/중/후 상태**로 기록되어야 한다.
  - 예: “2차함수 개념 부족” → “보강 중” → “해결됨” 또는 “추가 보강 필요”

## 4. 필요한 역할 정의

## 4.1 문제 출제자 (Problem Composer)
### 역할
- 문제를 만들고, 숙제 세트에 들어갈 문항을 구성한다.

### 책임
- 문제 텍스트/선지/정답/해설 초안 작성
- 문제 난이도 설정
- 단원/스킬 태깅
- 숙제 세트 목적 정의
  - 예: “2차함수 기본 정의 확인용”
  - 예: “중간고사 전 복습용”

### 중요한 기준
- 나중에 진도체크가 가능하도록 문제는 가능한 한 `skillId`, `nodeId`, `weakTag`와 연결되어야 한다.

---

## 4.2 배포 담당 (Railway Deployment Operator)
### 역할
- 출제된 문제/숙제 세트를 실제 운영 환경에서 사용 가능하게 만든다.

### 책임
- 숙제 JSON 또는 문제은행 데이터 정리
- Railway 반영 스크립트 실행
- 반영 결과 확인
- 배포 실패 시 재실행/수정

### 중요한 기준
- 출제 완료와 운영 반영은 분리되면 안 된다.
- “출제 완료”의 정의에는 최소한 다음이 포함되어야 한다.
  - 데이터 저장 완료
  - Railway 반영 완료 또는 반영 대기 상태 명확화

---

## 4.3 조교 (Teaching Assistant / Homework Operator)
### 역할
- 학생 숙제 제출 여부와 검토 상태를 운영 관점에서 관리한다.

### 책임
- 누가 숙제를 제출했는지 확인
- 미제출 학생 확인
- 제출물 검토 상태 관리
- 필요 시 반려/재제출 요청
- 숙제 완료 여부를 운영적으로 닫기

### 조교가 반드시 알아야 하는 질문
- 오늘 배정된 학생 중 누가 아직 제출하지 않았는가?
- 제출했지만 아직 검토되지 않은 숙제는 무엇인가?
- 반복 미제출 학생은 누구인가?
- 특정 학생이 최근 며칠 동안 얼마나 꾸준히 제출하고 있는가?

---

## 4.4 학생 상태 추적자 (Progress Monitor)
### 역할
- 학생이 특정 단원/스킬을 실제로 해결했는지 추적한다.

### 책임
- 약점 개념의 시작 상태 기록
- 숙제/오답/재도전 이후 상태 변화 기록
- 해결 여부 판단 기준 운영
- 교사/학부모/운영자에게 이해 가능한 형태로 요약

### 예시 질문
- 이 학생은 2차함수를 “모르는 상태”에서 “해결된 상태”로 넘어갔는가?
- 최근 2주 동안 같은 약점이 반복되고 있는가?
- 오답은 줄었지만 아직 개념 이해는 불안정한가?

---

## 4.5 PM / 운영 설계자
### 역할
- 위 흐름이 하나의 제품 루프로 돌아가도록 기준을 세운다.

### 책임
- 상태 정의 통일
- KPI 정의
- 어떤 시점에 조교 개입이 필요한지 규칙 정의
- 어떤 조건에서 “해결됨”으로 볼지 정책 정의

---

## 5. 필요한 화면 / 서비스 구조

## 5.1 문제 출제 화면 또는 출제 워크플로우
필수 요소:
- 문제 텍스트
- 문제 유형(객관식/주관식)
- 정답
- 해설
- 난이도
- 연결 단원/스킬
- 숙제 세트 포함 여부

운영적으로 중요한 것은 “문제를 만들었다”보다 **문제가 나중에 진단 가능한 구조를 가지는가**이다.

---

## 5.2 Railway 반영 워크플로우
필수 요소:
- 어떤 숙제 세트를 배포할지 선택
- 배포 실행
- 배포 성공/실패 로그
- 현재 운영 반영 상태 확인

권장 상태값:
- draft
- ready_to_publish
- publishing
- published
- failed

---

## 5.3 조교 대시보드
필수 섹션:
- 오늘 숙제 배정 현황
- 미제출 학생 목록
- 제출 완료 / 검토 대기 목록
- 반려 / 재제출 필요 목록
- 학생별 최근 제출 이력

핵심 목적:
- “누가 숙제를 안 냈는지”를 바로 알 수 있어야 한다.
- “지금 사람이 개입해야 하는 숙제”가 먼저 보여야 한다.

---

## 5.4 학생 진도체크 대시보드
필수 섹션:
- 핵심 약점 개념 목록
- 개념별 현재 상태
- 최근 숙제/문제 결과
- 오답 빈도 변화
- 해결 여부 판단

예시 상태값:
- 미진입
- 약점 확인
- 보강 중
- 재확인 필요
- 해결됨

예시 표현:
- “2차함수: 약점 확인 → 보강 중”
- “2차함수: 최근 3개 숙제에서 정답률 개선, 해결 가능성 높음”
- “2차함수: 계산 실수는 줄었지만 개념 질문은 아직 불안정”

## 7. 진도체크 정책 예시

## 7.1 왜 별도 진도체크가 필요한가
- 학생이 한 번 틀렸다는 사실은 문제의 시작점일 뿐이다.
- 실제 운영에서는 “아직 모름 / 좋아지고 있음 / 해결됨”의 구분이 필요하다.

## 7.2 예시 정책: 2차함수
초기 상태:
- 진단 또는 숙제 결과에서 2차함수 관련 문제를 반복 오답
- 상태 = `약점 확인`

중간 상태:
- 보강 숙제 2~3회 진행
- 계산형은 맞지만 개념형에서 흔들림
- 상태 = `보강 중`

해결 상태:
- 최근 N회 관련 문항에서 일정 수준 이상 정답
- 개념형/적용형 모두 최소 기준 충족
- 상태 = `해결됨`

재확인 상태:
- 한동안 맞다가 다시 반복 오답 발생
- 상태 = `재확인 필요`

## 7.3 해결 판단 기준 예시
- 최근 3회 중 2회 이상 관련 과제 통과
- 개념형 문항 정답률 80% 이상
- 동일 weakTag 반복 발생 없음

이 기준은 나중에 데이터 기반으로 조정 가능해야 한다.

## 8. KPI 제안

### 8.1 출제/배포 운영 KPI
- 출제 후 배포 완료까지 걸린 시간
- 배포 실패율
- 숙제 세트 운영 반영률

### 8.2 조교 운영 KPI
- 숙제 제출률 (24h)
- 미제출 추적 후 제출 전환율
- 검토 완료까지 평균 시간
- 반려 후 재제출 완료율

### 8.3 진도체크 KPI
- 약점 개념 해결률
- 재도전 개선률
- weakTag 반복 감소율
- 특정 핵심 단원(예: 2차함수) 해결 전환율

## 9. 우선순위 제안

### Phase 1 — 운영 최소 루프 완성
가장 먼저 필요한 것:
- 문제 출제 구조 정리
- Railway 반영 절차 정리
- 조교가 제출 여부를 확인할 수 있는 화면/리스트

이 단계 목표:
- “출제했는데 운영에 반영 안 됨” 문제 제거
- “누가 숙제를 안 냈는지 모름” 문제 제거

### Phase 2 — 검토와 피드백 운영 강화
- 제출 검토 상태 세분화
- 반려/재제출 루프 명확화
- 조교 피드백 기록 구조화

### Phase 3 — 학생 상태 추적 체계 강화
- 개념별 상태값 도입
- 약점 해결 여부 추적
- 교사/학부모용 진도 요약 강화

## 10. 제품적으로 중요한 해석

이 서비스에서 중요한 것은 단순히 문제를 잘 만드는 것이 아니다.

진짜 중요한 것은:
- 문제를 바로 운영에 연결하고,
- 학생이 제출했는지 놓치지 않으며,
- 특정 개념이 해결되었는지 계속 추적하고,
- 그 결과가 다음 숙제와 추천으로 연결되는 것이다.

즉, `calculate_math`의 운영 체계는
**콘텐츠 운영 + 숙제 운영 + 사람 운영(조교) + 학습 상태 운영**
이 함께 돌아가야 완성된다.

4. 학생 진도체크에 `약점 확인/보강 중/재확인 필요/해결됨` 상태 도입
5. 핵심 단원(예: 2차함수)부터 해결 여부 추적 정책을 시범 적용

이 다섯 가지가 정리되면, 이후 PM/BE/FE/QA/Research 역할 정의도 훨씬 선명해진다.
