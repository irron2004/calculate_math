# 수학 모험 역할 및 Agent R&R 정의 문서

## 0. 문서 목적

이 문서는 `calculate_math`의 메인 컨셉인 **수학 모험(Math Adventure)** 을 실제 제품과 운영 체계로 만들기 위해,
어떤 역할이 어떤 책임을 지고 어떤 산출물을 만들어야 하는지 정의한다.

이 문서는 단순한 직무 소개 문서가 아니라,
**운영 기준 문서**다.

## 1. 수학 모험의 제품 정의

### 1.1 한 줄 정의
**수학 모험은 덧셈에서 시작해 미적분까지 이어지는, 선수학습 기반의 그래프형 수학 커리큘럼이다.**

### 1.2 핵심 구조
- 각 수학 개념은 하나의 **노드(node)** 다.
- 노드 사이에는 선수학습 관계와 다음 학습 경로가 **엣지(edge)** 로 연결된다.
- 학습자는 쉬운 개념부터 시작해 점차 더 깊은 개념으로 이동한다.

예:
- 1자리 덧셈 → 2자리 덧셈 → 3자리 덧셈
- 사칙연산 → 식 세우기 → 방정식 → 함수 → 미분 → 적분

### 1.3 상위 트랙
수학 모험의 메인 트랙은 2022 교육과정의 4영역을 기준으로 구성한다.

- **수와 연산(NA)**: 1자리 덧셈 → 2자리 덧셈 → ... → 방정식 → 함수 → 미적분
- **변화와 관계(RR)**: 규칙 찾기 → 비례 → 함수 → 미적분 (상위에서 수와연산과 합류)
- **도형과 측정(GM)**: 점/선/면 → 도형 → 넓이 → 좌표기하 → 벡터
- **자료와 가능성(DP)**: 세기 → 표/그래프 → 평균 → 확률 → 확률분포

### 1.4 제품이 반드시 보여줘야 하는 것
- 지금 학습자가 어디에 있는가
- 다음에 무엇을 배워야 하는가
- 왜 그 노드가 다음 단계인가
- 이 트랙을 계속 따라가면 어디까지 갈 수 있는가

## 2. 조직 구조: 서비스 조직과 Research 조직

### 2.1 왜 두 조직으로 나누는가

기존 문서에서는 모든 역할이 하나의 순차 루프 안에 있었다:
그래프 설계 → 교육 검증 → 문제 출제 → 구현 → 배포 → 숙제 배정.

이 구조의 문제는 **숙제를 하나 내려면 그래프 설계부터 배포까지 전체를 돌아야 한다**는 것이다.
시뮬레이션 결과, 숙제 하나 내는 데 4~5일이 걸리고, 한 사람이 7개 역할을 전환하며 작업해야 했다.
Graph Architect에 초기 병목이 집중되고, BE에 구현이 쏠리며, Research와 운영이 서로를 블로킹했다.

실제로는:
- 숙제 운영(문제 출제·배정·채점·스케줄)은 **이미 Beta 서비스 중**이다.
- 커리큘럼 그래프(노드 연결·추천·진단)는 **아직 연구 단계**다.

두 작업은 성격과 속도가 다르며, 하나가 다른 하나를 블로킹할 이유가 없다.
따라서 역할을 **서비스 조직**과 **Research 조직**으로 분리한다.

### 2.2 두 조직의 정의

**서비스 조직** — 학생에게 숙제를 내고, 채점하고, 스케줄을 관리하는 운영 조직.
그래프가 없어도 동작한다. 지금 돌아가고 있는 Beta 서비스가 이 조직의 산출물이다.

**Research 조직** — 서비스 조직에서 쌓인 데이터(출제 이력, 오답 패턴, 학생 피드백)를 관찰하고,
커리큘럼 그래프의 노드·엣지를 설계하는 연구 조직.
서비스 조직의 속도를 늦추지 않으면서, 그래프 구조를 점진적으로 만들어간다.

### 2.3 두 조직의 연결점

두 조직은 **노드 이름(단원명)**으로 연결된다.

- 서비스 조직은 문제를 출제할 때 단원명을 태깅한다 (예: "로그 성질", "판별식").
- Research 조직은 그 단원명을 그래프 노드로 정의하고 선수관계를 연결한다.
- 나중에 노드 이름 매핑이 완료되면, 기존 문제·숙제 데이터가 자연스럽게 그래프에 연결된다.

이 연결 작업이 잘 이뤄지기 위해, **단원명 네이밍 규칙**을 두 조직이 공유해야 한다.

### 2.4 두 조직의 시나리오

**서비스 조직 시나리오 (매일 반복)**
```
학생 피드백 (예: "민수가 판별식 문제를 못 풀어")
  → 관련 단원 분석 (로그 성질, 판별식)
  → DB 조회 / 없으면 생성
  → 관련 문제 출제
  → 숙제 배정 · 채점 · 스케줄 관리
```

**Research 조직 시나리오 (주기적)**
```
서비스 조직에서 쌓인 데이터 관찰
  → 단원 간 관계 분석 (예: "로그는 2차함수, 방정식과 관련이 있다")
  → 학생에게 출제된 문제들 조회
  → 노드 연결관계 분석 · 그래프 반영
  → 선수학습 관계 · mastery 기준 · 진단 규칙 설계
```

### 2.5 역할 설계 원칙

역할은 직함이 아니라 **책임 단위**다.
초기에는 한 사람이 여러 역할을 맡을 수 있지만, 문서상 책임은 분리되어야 agent에게 명확히 위임할 수 있다.

각 역할은 다음 네 가지가 명확해야 한다:
1. 무엇을 결정하는가
2. 무엇을 만드는가
3. 무엇을 입력으로 받는가
4. 누구에게 어떤 출력물을 넘기는가

agent는 역할을 대체하는 것이 아니라, **역할이 정한 목표와 기준을 실행하는 작업 단위**다.

## 5. 서비스 조직 상세 R&R

## 5.1 Commander / Service Owner

### 역할 목적
수학 모험의 핵심 방향을 결정하고, 무엇을 제품의 본질로 볼지 최종 승인하는 역할이다.

### 책임
- 수학 모험의 한 줄 정의 승인
- 상위 트랙 우선순위 결정
- MVP 범위 확정
- 역할 간 충돌 시 최종 판단

### 입력
- PM의 phase plan
- Graph Architect의 트랙 구조안
- Researcher의 교육적 타당성 검토

### 출력
- 승인된 제품 원칙
- 트랙 출시 순서
- MVP 완료 기준

### agent에게 넘길 수 있는 작업 예시
- 메인 컨셉 문구 초안 비교
- 트랙 출시 우선순위 옵션 정리
- MVP 범위 시나리오 비교

## 5.2 PM / Orchestrator

### 역할 목적
서비스 조직의 운영을 실행 가능한 단계로 나누고, 역할 간 작업 순서를 조정하는 역할이다.

### 책임

- 백로그 정리
- 역할 간 handoff 기준 문서화

- Content/조교의 운영 현황
- BE/FE/QA의 구현 제약사항

### 출력
- roadmap
- phase backlog
- 역할별 task packet
- weekly priorities

### agent에게 넘길 수 있는 작업 예시
- MVP 범위를 epic과 task로 분해
- 역할별 산출물 체크리스트 생성
- 주간 실행 순서표 작성

- 학생 피드백 (예: "민수가 판별식 못 풀어")

- 문제은행
- 숙제 세트
- 단원명 태그 목록 (Research 조직의 입력이 됨)

## 6. Research 조직 상세 R&R

## 6.1 Curriculum Graph Architect

### 역할 목적
서비스 조직에서 쌓인 단원명·문제·피드백 데이터를 분석하여 커리큘럼 그래프의 노드와 엣지를 설계하는 역할이다.

### 책임
- 수와연산(NA) / 변화와관계(RR) / 도형과측정(GM) / 자료와가능성(DP)의 상위 구조 정의
- 서비스 조직의 단원명을 그래프 노드로 정의
- 선수학습 관계 정의
- 트랙의 장기 경로 정의 (덧셈 → 미적분)
- unlock 조건 정의

### 핵심 질문
- 서비스에서 태깅된 단원들 사이에 어떤 선수관계가 있는가?
- 어떤 개념이 하나의 노드가 되어야 하는가?
- 이 트랙이 어떻게 미적분까지 자연스럽게 이어지는가?

### 입력
- 서비스 조직의 단원명 태그 목록
- 서비스 조직의 출제 이력 · 오답 패턴
- 교육과정 기준 자료
- Data Analyst의 단원 관계 분석

### 출력
- track map
- node taxonomy
- prerequisite edge spec
- start node / milestone node 정의

### agent에게 넘길 수 있는 작업 예시
- 서비스에서 태깅된 단원 목록 → 노드 초안 변환
- 노드 granularity 재조정안 비교
- prerequisite conflict 탐지

## 6.2 Education Researcher / Learning Designer

### 역할 목적
그래프가 교육적으로 성립하는지 검증하고, 각 노드가 어떤 학습목표를 가져야 하는지 정의하는 역할이다.

### 책임
- 노드별 학습목표 정의
- mastery 기준 정의
- 대표 오개념 정리
- drill-down 범위 정의

### 입력
- Graph Architect의 노드/엣지 구조
- 서비스 조직의 오답 데이터
- 교육과정 자료

### 출력
- node learning objective spec
- mastery criteria
- misconception list

### agent에게 넘길 수 있는 작업 예시
- 특정 노드의 mastery 정의
- 함수 진입 전 필수 선수개념 검토
- 트랙의 단계 간 비약 여부 검토

## 6.3 Learning Diagnosis Designer

### 역할 목적
서비스 조직에서 관찰된 오답 패턴을 해석하고, 그래프 상에서 어떤 보강 경로로 연결할지 정의하는 역할이다.

### 책임
- 오답 원인 taxonomy 설계
- 상태 전이 정의 (약점 확인 → 보강 중 → 해결됨)
- 약점 노드 식별 규칙 정의
- "해결됨" 판정 기준 정의

### 입력
- 서비스 조직의 오답 데이터
- Graph Architect의 노드 구조
- Education Researcher의 학습목표

### 출력
- diagnosis taxonomy
- state transition rules
- remediation path rules

### agent에게 넘길 수 있는 작업 예시
- 특정 단원 오답 taxonomy 작성
- 진단 상태 전이표 초안
- 약점 노드 재추천 규칙 설계

## 6.4 Data Analyst

### 책임
- 출제 이력에서 단원 간 동시출현 패턴 분석
- 오답 반복 패턴 분석
- 학생별 학습 경로 분석
- 개선 우선순위 제안

### 입력
- 서비스 조직의 문제/숙제/제출 데이터
- 서비스 조직의 학생 피드백

### 출력
- 단원 관계 분석 리포트
- bottleneck analysis
- next-iteration proposal

### agent에게 넘길 수 있는 작업 예시
- 특정 학생의 출제 문제 조회 및 단원 관계 추출

## 7. 역할 간 handoff 표준

agent에게 작업을 넘길 때는 아래 형식을 기본으로 사용한다.

### 7.1 handoff packet 필수 항목
- 역할명
- 이번 작업의 목적
- 입력 문서 / 데이터
- 결정해야 하는 질문
- 출력 형식
- 완료 조건
- 다음으로 넘길 대상 역할

### 7.2 좋은 handoff의 예
- "수와연산 트랙의 시작 노드 10개를 정의하라" 보다
- "초등 연산에서 중등 식 세우기로 넘어가기 전까지의 시작 노드 10개를 정의하고, 각 노드에 prerequisite와 mastery 기준을 붙여 Graph Architect 출력 형식 표로 제출하라"가 좋다.

### 7.3 agent handoff packet 템플릿

```md
## Role
- Curriculum Graph Architect

## Task
- 수와연산 트랙의 시작 노드 10개 정의

## Goal
- 초등 연산에서 문자식 진입 전까지의 입문 경로를 구조화한다.

## Inputs
- 서비스 조직의 단원명 태그 목록
- 기존 그래프 규칙 문서
- 교육과정 기준 메모

## Questions to Resolve
- 어떤 노드를 시작 노드로 둘 것인가?
- 어떤 prerequisite를 필수로 둘 것인가?
- 어떤 노드를 milestone로 볼 것인가?

## Output Format
- node id / label / prerequisite / mastery / milestone 여부 표

## Done When
- 시작 노드 10개가 표로 정리되어 있고
- 각 노드에 prerequisite와 mastery가 붙어 있으며
- 다음 handoff 대상이 명시되어 있다.

## Next Handoff
- Education Researcher
- Backend Engineer
```

## 8. 실행 단계

### 8.1 현재 상태
- **서비스 조직**: Beta 운영 중. 숙제 배정·채점·스케줄 관리가 동작하고 있다.
- **Research 조직**: 연구 단계. 그래프 구조와 노드 연결 설계가 진행 중이다.

### 8.2 서비스 조직 우선 과제
서비스 조직은 그래프 완성을 기다리지 않고 독립적으로 운영을 강화한다.

1. 학생 피드백 → 단원 분석 → 문제 출제 루프 고정
2. 숙제 배정·제출·검토 플로우 안정화
3. 문제 출제 시 단원명 태깅 규칙 정립

### 8.3 Research 조직 우선 과제
Research 조직은 서비스 조직의 데이터를 관찰하며 그래프를 점진적으로 설계한다.

1. 서비스에서 태깅된 단원명 목록 수집 및 노드 후보 정리
2. 노드 간 선수관계 초안 설계
3. 서비스 데이터(출제 이력, 오답 패턴)를 기반으로 관계 검증

### 8.4 합류 시점
두 조직의 산출물이 합쳐지는 시점은 다음 조건이 충족될 때다:

- Research 조직에서 핵심 트랙의 노드·엣지가 안정화됨
- 서비스 조직의 단원명과 Research 조직의 노드 이름 매핑이 완료됨
- 이 매핑을 통해 기존 문제·숙제 데이터가 그래프에 연결 가능해짐

## 9. 결론

### 9.1 왜 이렇게 재정의했는가

기존 문서는 모든 역할을 하나의 순차 루프에 배치했다.
이 구조에서는 **숙제를 하나 내는 데 그래프 설계부터 배포까지 전체를 돌아야** 했다.

시뮬레이션 결과 세 가지 문제가 드러났다:
1. **Graph Architect 병목**: 거의 모든 역할이 그래프 설계 산출물을 기다림
2. **역할 전환 비용**: 한 사람이 7개 역할을 컨텍스트 스위칭하며 작업
3. **서비스와 연구의 상호 블로킹**: 그래프가 없으면 숙제를 못 내고, 숙제 데이터가 없으면 그래프를 못 만드는 교착

해결: 숙제 운영(서비스)과 그래프 설계(Research)를 **독립된 루프**로 분리했다.
서비스 조직은 그래프 없이도 숙제를 내고, Research 조직은 서비스 데이터를 관찰하며 그래프를 만든다.
두 루프는 **단원명(노드 이름)**으로 나중에 합류한다.

### 9.2 핵심 원칙

1. 서비스 조직은 Research 조직을 기다리지 않는다.
2. Research 조직은 서비스 조직의 데이터를 입력으로 사용한다.
3. 두 조직의 연결점은 단원명 네이밍 규칙이다.
4. agent는 명확한 역할 정의 위에서 실행력을 제공하는 수단이다.
