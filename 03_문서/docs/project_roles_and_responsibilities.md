# calculate_math 프로젝트 역할 정의 문서

## 0. 문서 목적
이 문서는 `calculate_math` 프로젝트를 운영하고 확장하기 위해 필요한 역할을 정의한다.

이 프로젝트는 단순한 문제풀이 서비스가 아니라,

- 수학 교육과정의 단원/스킬을 그래프로 구조화하고,
- 문제를 그 그래프와 연결하며,
- 숙제를 운영 환경에 배포하고,
- 학생 제출 여부를 확인하고,
- 특정 개념이 실제로 해결되었는지 추적하는

학습 운영 시스템이다.

따라서 필요한 역할도 일반적인 기능 개발 역할만으로는 충분하지 않다.

---

## 1. 기본 원칙

### 1.1 역할은 “직함”보다 “책임 단위”로 본다
초기 팀에서는 한 사람이 여러 역할을 동시에 맡을 수 있다.

예:
- 한 사람이 `교육과정 그래프 설계자 + 문제 출제자 + PM`을 맡을 수 있다.
- 한 사람이 `백엔드 + 배포 운영자`를 맡을 수 있다.
- 한 사람이 `프론트엔드 + QA`를 맡을 수 있다.

중요한 것은 사람 수보다도, **이 프로젝트가 돌아가기 위해 어떤 책임이 반드시 존재해야 하는가**를 명확히 하는 것이다.

### 1.2 역할은 학습 루프 기준으로 정의한다
이 프로젝트의 핵심 루프는 다음과 같다.

1. 교육과정 그래프 설계
2. 문제 출제 및 태깅
3. 숙제 구성 및 배포
4. 학생 제출 및 검토
5. 오답/진도 해석
6. 다음 추천/다음 숙제 생성

필요한 역할은 모두 이 루프의 어느 구간을 책임지는지로 정의해야 한다.

## 2. 핵심 역할 정의

## 2.1 교육과정 그래프 설계자

### 역할 설명
수학 교육과정의 각 단원과 세부 스킬을 그래프 노드로 정의하고, 선수학습 관계와 다음 학습 경로를 설계하는 역할이다.

### 왜 중요한가
이 프로젝트의 차별점은 “어떤 단원이 다음에 와야 하는가”를 감으로 추천하는 것이 아니라,
**수학 개념 간의 구조적 관계를 그래프로 표현하는 것**에 있다.

예:
- 등차수열 노드는 덧셈/뺄셈/곱셈과 선수관계로 연결될 수 있다.
- 특정 단원을 이해하려면 어떤 AtomicSkill이 선행되어야 하는지 정의할 수 있다.

### 책임
- `CourseStep` 정의
- `AtomicSkill` 정의
- `requires`, `prepares_for`, `enables` 관계 설계
- 단원과 세부 스킬 연결
- 오개념/약점 개념 연결의 기준 정의

### 주요 산출물
- 교육과정 그래프 문서
- 노드/엣지 정의표
- 선수학습 규칙표
- 추천 경로 규칙

---

## 2.2 문제 출제자 / 콘텐츠 설계자

### 역할 설명
그래프에 연결될 문제를 만들고, 각 문제가 무엇을 측정하는지 정의하는 역할이다.

### 왜 중요한가
그래프가 학습 지도라면, 문제는 학생의 현재 상태를 읽어내는 측정 도구다.
문제가 그래프와 연결되지 않으면 진단과 추천이 약해진다.

### 책임
- 문제 작성
- 정답/해설 작성
- 난이도 설정
- 문제와 `nodeId` / `skillId` 연결
- `weakTag` 정의
- 숙제 세트 구성

### 주요 산출물
- 문제은행
- 문제 메타데이터
- 숙제 세트
- 해설 초안

---

## 2.3 학습 진단 설계자

### 역할 설명
학생이 왜 틀렸는지, 특정 개념이 어느 상태인지 정의하고 진단 체계를 설계하는 역할이다.

### 왜 중요한가
이 프로젝트의 핵심은 정답률 자체보다,
**왜 틀렸는지**와 **이 개념이 해결되었는지**를 설명 가능한 형태로 다루는 것이다.

### 책임
- 오답 원인 분류 체계 설계
- 진도 상태값 정의
- 해결 여부 판단 기준 설계
- weakTag와 개념 상태 연결

### 예시 상태
- 약점 확인
- 보강 중
- 재확인 필요
- 해결됨

### 주요 산출물
- 진단 taxonomy
- 상태 전이 규칙
- 해결 판단 정책
- 리포트용 해석 문구 기준

---

## 2.4 PM / 학습 운영 설계자

### 역할 설명
제품이 어떤 순서로 만들어지고 운영될지 결정하는 역할이다.

### 왜 중요한가
이 프로젝트는 그래프, 문제, 숙제, 조교, 진도체크가 모두 얽혀 있어서
무엇을 먼저 만들고 어떤 상태를 성공으로 볼지 정리하는 역할이 반드시 필요하다.

### 책임
- 제품 우선순위 결정
- MVP 범위 정의
- 상태값 표준화
- KPI 정의
- 각 역할 간 책임 경계 정의

### 주요 산출물
- PRD
- 우선순위 문서
- 운영 정책
- KPI 문서

---

## 2.5 백엔드 엔지니어

### 역할 설명
그래프, 추천, 진단, 숙제, 제출, 진도 상태를 저장하고 연결하는 서버 구조를 만드는 역할이다.

### 왜 중요한가
이 프로젝트의 백엔드는 단순 CRUD가 아니라,
교육과정 그래프와 숙제 운영, 학생 상태 추적을 한 시스템 안에서 묶어야 한다.

### 책임
- 그래프 데이터 모델 설계
- 추천 API 구현
- 진단 API 구현
- 숙제/제출/검토 API 구현
- 학생 상태/진도 API 구현

### 주요 산출물
- DB schema
- backend API
- 추천/진단 로직
- 운영용 상태 저장 구조

---

## 2.6 프론트엔드 엔지니어

### 역할 설명
학생, 조교, 관리자에게 필요한 학습 화면과 운영 화면을 구현하는 역할이다.

### 왜 중요한가
이 서비스는 알고리즘이 좋아도,
사용자가 “왜 이걸 해야 하는지”를 화면에서 이해하지 못하면 가치가 약해진다.

### 책임
- 학생 홈 화면 구현
- 숙제 제출 화면 구현
- 추천 이유 노출 UI 구현
- 조교 대시보드 구현
- 진도체크/리포트 화면 구현
- 그래프 시각화 구현

### 주요 산출물
- 학생용 UI
- 조교용 UI
- 관리자용 UI
- 리포트/그래프 화면

---

## 2.7 QA / 학습 흐름 검증자

### 역할 설명
기능이 동작하는지뿐 아니라 학습 루프가 끊기지 않는지 검증하는 역할이다.

### 왜 중요한가
이 프로젝트는 “버튼이 눌리는가”보다,
숙제 배정 → 제출 → 검토 → 진도 반영 → 다음 추천 흐름이 제대로 연결되는지가 더 중요하다.

### 책임
- 학습 흐름 시나리오 테스트
- 상태값 일관성 검증
- 추천 근거/진단 결과 노출 검증
- 회귀 테스트
- 운영 이슈 재현 및 기록

### 주요 산출물
- QA 체크리스트
- E2E 시나리오
- 회귀 테스트 결과
- 이슈 리포트

---

## 2.8 배포 / 플랫폼 운영자 (DevOps)

### 역할 설명
출제된 숙제와 시스템 변경사항을 실제 운영 환경(Railway)에 반영하고 안정적으로 유지하는 역할이다.

### 왜 중요한가
이 프로젝트는 실제 운영 중인 숙제/문제 흐름과 연결되기 때문에,
배포는 개발의 마지막 단계가 아니라 운영의 핵심 단계다.

### 책임
- Railway 배포
- 운영 환경 설정
- 배포 로그 확인
- 장애 대응
- 배포 체크리스트 관리

### 주요 산출물
- 배포 파이프라인
- 운영 설정 문서
- 배포 로그/장애 기록
- 운영 체크리스트

---

## 2.9 조교 / 숙제 운영자

### 역할 설명
학생의 숙제 제출 여부와 검토 상태를 실제 운영 관점에서 관리하는 역할이다.

### 왜 중요한가
학습 서비스에서는 학생이 숙제를 냈는지, 검토가 끝났는지, 재제출이 필요한지가 매우 현실적인 운영 핵심이다.

### 책임
- 미제출 학생 확인
- 제출 완료 학생 확인
- 검토 대기 숙제 정리
- 반려/재제출 상태 관리
- 필요 시 운영 메시지 전달

### 주요 산출물
- 제출 현황판
- 미제출 목록
- 검토 대기 목록
- 반려/재제출 기록

---

## 2.10 학생 상태 추적자 / 교육 리서처

### 역할 설명
학생이 특정 개념을 실제로 해결했는지, 약점이 개선되고 있는지를 분석하는 역할이다.

### 왜 중요한가
이 역할이 있어야 서비스가 단순 숙제 플랫폼이 아니라,
학생이 실제로 나아지고 있는지를 해석하는 학습 시스템이 된다.

### 책임
- 개념별 진도 상태 추적
- weakTag 변화 관찰
- 특정 단원 해결 여부 분석
- 추천/진단 정책 개선 피드백 제공

### 주요 산출물
- 학생 상태 리포트
- 약점 개선 분석
- 개념 해결 여부 판단 리포트
- 정책 개선 제안

## 3. 역할 간 관계

### 3.1 그래프 → 문제 → 운영 흐름
- 교육과정 그래프 설계자는 학습 구조를 만든다.
- 문제 출제자는 그 구조에 맞는 문제를 만든다.
- 학습 진단 설계자는 문제 결과를 해석하는 기준을 만든다.

### 3.2 제품 → 구현 흐름
- PM은 우선순위와 상태 기준을 정한다.
- 백엔드는 데이터 구조와 API를 만든다.
- 프론트엔드는 학생/조교/관리자 화면을 만든다.

### 3.3 운영 → 개선 흐름
- 조교는 제출과 검토 상태를 관리한다.
- 학생 상태 추적자는 개념 해결 여부를 본다.
- QA는 전체 학습 루프가 끊기지 않는지 검증한다.
- DevOps는 이 흐름이 실제 운영 환경에서 안정적으로 돌아가게 만든다.

## 4. 초기 단계에서 필요한 최소 역할 조합

## 4.1 가장 먼저 필요한 역할
- 교육과정 그래프 설계자
- 문제 출제자
- PM / 운영 설계자
- 백엔드 엔지니어
- 프론트엔드 엔지니어
- 배포 / Railway 운영자

이 여섯 역할이 있어야 최소한의 제품/운영 루프가 돌아간다.

## 4.2 빠르게 보강되어야 하는 역할
- 조교 / 숙제 운영자
- QA / 학습 흐름 검증자

이 둘이 붙어야 실제 운영에서 “누가 숙제를 안 냈는지”, “무엇이 끊겼는지”를 잡아낼 수 있다.

## 4.3 성장 단계에서 더 중요해지는 역할
- 학생 상태 추적자 / 교육 리서처
- 데이터 분석 담당

이 단계부터는 단순 운영을 넘어서, 진짜 학습 효과를 해석하는 일이 중요해진다.

## 6. 제품 관점의 해석

이 프로젝트에서 필요한 역할을 일반적인 소프트웨어 팀처럼만 보면,
`PM / BE / FE / QA` 정도로 끝날 수 있다.

하지만 `calculate_math`는 다음과 같은 프로젝트다.

- 교육과정 구조를 그래프로 정의해야 하고,
- 문제를 그 그래프에 연결해야 하며,
- 숙제를 실제 운영 환경에 반영해야 하고,
- 학생 제출 여부를 놓치지 않아야 하며,
- 특정 개념이 해결되었는지 지속적으로 추적해야 한다.

따라서 이 프로젝트의 진짜 핵심 역할은 다음 다섯 축이다.

1. 교육과정 그래프를 설계하는 역할
2. 문제를 그래프에 연결하는 역할
3. 숙제를 배포하고 운영하는 역할
4. 제출 여부를 관리하는 조교 역할
5. 학생 상태를 추적하고 해석하는 역할

이 다섯 축이 살아 있어야, 나머지 PM/BE/FE/QA도 올바른 방향으로 작동한다.
